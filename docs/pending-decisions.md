## `ai_mode`をCPU席ごとに選べるようにする案

フェーズ2（CPU混在・6人フル対応）の実装時に検討し、スコープを絞るため見送った。

- **内容**：現状、実プレイのCPU席は`smart`ロジック固定（`decideAiAction`のデフォルト引数）。これを部屋ごと、あるいはCPU席ごとに`smart`/`naive`を選べるようにする案
- **見送った理由**：`players`テーブルへの列追加とEdge Functions側でのper-seat lookupが必要になり、フェーズ2のスコープが広がるため
- **再検討のトリガー**：実際に友人と遊んでみて「もっと弱いCPUが欲しい」「歯ごたえのあるCPUが欲しい」等の要望が出た場合。`naive`は本来シミュレーション比較用のベースラインであり、そのまま「弱いCPU」に転用してよいかは別途検討が必要（`ai.ts`のコメント通り「状況を一切見ず固定確率で予約する」だけなので、単に弱いのではなく「支離滅裂」に近い挙動になる点に注意）

## `smart` AIロジックの改善余地

`src/engine/ai.ts`の`decideAiAction`を読んだ上での所見（2026-07-25時点）。実際に改善が必要かはシミュレーション結果次第であり、あくまで「気付いた観点」の記録。

現在の`smart`ロジックの中身：

- **予約判断**：自分の`chipScore`が他プレイヤーの平均より高い（＝負けている）か、黒チップ保有数、今出せるカードの割合（`canPlay.length / hand.length`）、ペナルティ値に応じた閾値（`0.2 + penalty/10 * 0.3`）の組み合わせで判断
- **カードプレイ**：出せるカードの中から「手札内で他に同じ値がない（ユニークな）カード」を最優先、次に数値が大きいカードを優先する貪欲法
- **山札を引くか**：手札が少ないほど引く確率が高くなる単純な確率式（`max(0.15, 1 - 手札枚数*0.09)`）で、山札の残り枚数や他プレイヤーの状況は見ていない

気付いた改善余地（未検証の仮説）：

1. 予約判断が「自分 vs 他プレイヤー平均」の一点比較のみで、2位以下との差やチップ合計が40点（敗北ライン）にどれだけ近いかという終盤リスクを見ていない
2. カードプレイの「ユニーク優先」ヒューリスティックが本当に有効か未検証（同じ値の重複カードを温存する動きになるが、これが得なのかは`CvcStats`で未検証）
3. 山札を引く確率が自分の手札枚数だけに依存しており、山札の残り枚数（切れが近いか）や場の状況を見ていない
4. 各種閾値（`0.2 + penalty/10*0.3`、黒チップ2枚以上で確実に予約、1枚なら60%等）はいずれも初期実装時の勘に基づく値で、`CvcStats`での体系的なパラメータ探索を経ていない

再検討のトリガー：`CvcStats`（CvC統計シミュレーション）で`smart` vs `naive`の比較データが十分に溜まった段階、または実プレイでCPUの挙動に違和感が出た場合。

## 将来モジュール候補：メカニクス4・15

CLAUDE.mdに一行で記録されていた将来検討候補を、説明を補って転記したもの。番号の由来（外部の分類表等）は本リポジトリ内に記録がなく、CLAUDE.mdに書かれていた通しの呼び名をそのまま踏襲している。

- **メカニクス4：手札の可視性が時間で変化**
  - 現状ビクーニャでは「自分の手札は常に自分だけに見える／他人の手札は常に非公開（枚数のみ公開）」という固定ルール。このメカニクスは、ラウンド経過や特定のトリガー（一定ターン数経過、特定カードのプレイ等）に応じて、手札の一部または全部が一時的に公開される／逆に一時的に見えなくなる、といった「可視性そのものが時間軸で変化する」仕組みを指す
  - ビクーニャへの適用イメージ（未検討・要ブレスト）：例えば「予約宣言した瞬間、その人の手札の枚数だけでなく最大値カードが一時的に公開される」等、駆け引きの緊張感を高める方向での応用が考えられる
- **メカニクス15：隠れた役割/正体**
  - 各プレイヤーに非公開の役割やチーム、勝利条件が割り当てられ、ゲーム中に徐々に明かされる（あるいは最後まで隠され続ける）タイプのメカニクス（人狼系ゲーム等でよく使われる）
  - ビクーニャへの適用イメージ（未検討・要ブレスト）：予約者に対して「実は妨害役」等の裏目的を持たせる、といった拡張モジュール案が考えられるが、現状のシンプルな得点最小化ゲームとの整合性（誰が何を目指すゲームになるか）は未整理

再検討のトリガー：基本ルール（解釈B）でのバランス調整が一段落し、追加モジュールとしての実装リソースを割ける段階になった時。どちらも現時点ではアイデア段階に過ぎず、具体的なルール文章・UI設計は未着手。
