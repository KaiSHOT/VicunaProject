# PvPオンライン化・リリース要件（決定事項ログ）

作成日: 2026-07-20

このドキュメントは「ネット公開・PvP同時対戦化」に向けた要件を、ユーザーとの
一問一答で確定した内容として記録するもの。実装が進むにつれて内容が古くなる
可能性があるため、変更があれば都度このファイルを更新すること。

## Goal

1. このプロジェクトをリリースしてネット公開する。
2. PvPで最大人数が同時に接続して遊べるようにする。
3. PvPに対して、合計最大数までの任意の数のCPUを追加できるものとする。
4. プレイヤーネームなどは一時的に記憶しておくものとする。
5. プレイ履歴は保存してのちの学習に使える形にする。
6. 実行に最適な画面を作成する。
7. カードなどは実際にプレイする際に近い形で作成を行う。
8. 著作権には気を付けつつ、作成を行う。

## Test

1. 画面描画を含めて見切れがないように網羅的にテストを行う。
2. スマートフォンにも対応する。

## 確定事項（一問一答で合意した内容）

| 論点 | 決定内容 |
| --- | --- |
| リリース先 | Vercel（Hobby、広告なし方針）。デプロイ対象ブランチは`main` |
| PvP通信基盤 | Supabase Realtime（マネージド。自前WebSocketサーバーやPartyKit/Colyseusは不採用） |
| ゲームロジック実行主体 | **サーバー権威**。`src/engine/rules.ts`をSupabase Edge Functions上で実行し、クライアントは表示専用。不正操作を防ぐため |
| 部屋（ルーム）の参加フロー | ルームコードを発行 → URL/コードを友人に共有して参加。公開ロビー一覧方式は不採用 |
| PvPルームの最大人数 | 人間+CPU合計6人（現行PvC＝人間1人+AI1〜5人の構成を踏襲） |
| プレイヤーネーム | ログインなし。ニックネーム＋ルームコードのみで識別。端末をまたいだ永続化はしない |
| プレイ履歴の保存範囲 | サーバー集約（開発者のバランス調整・AI改善用）。個人の振り返り用途ではなく、`CvcStats`と同じ視点を実戦データで補強する位置づけ |
| プレイ画面 | PvP専用の新しい画面を新規設計（`PvcGame.tsx`のレイアウト踏襲ではなく作り直す） |
| カード意匠 | 外部イラスト（発注 or 生成AI）で本格的な絵柄を新規作成。LAMA等の既存カードゲームの絵柄・ロゴは模倣しないことが前提 |
| 網羅テストの方法 | Playwright等でE2E／ビジュアルリグレッションを新規導入（現状テスト基盤は未導入だったため今回が初導入） |
| モバイル対応の範囲 | レスポンシブWebのみ。PWA化（ホーム画面追加・オフライン対応）は対象外 |

## 導かれる技術スタック（暫定）

- フロントエンド: 既存 Vite + React + TypeScript + Tailwind CSS を継続
- バックエンド: Supabase（Realtime + Postgres + Edge Functions）を新規導入
- ゲームロジック: `src/engine/`をEdge Functions（Deno環境）上でも動かせる形に整理・移植が必要（Node依存がなければ大きな書き換えは不要な見込みだが要検証）
- テスト: Playwright新規導入
- ホスティング: Vercel（フロント）＋ Supabase（バックエンド）

## バックエンド基盤の比較検討（Supabase確定の経緯）

PvP通信基盤について、Supabase以外にGCP（Firebase）、AWS（Amplify Gen2 + Aurora Serverless Postgres）、Azure（SignalR + Functions + Postgres）、Cloudflare（Durable Objects/PartyKit）、自前Node WSサーバーを比較検討した。詳細な比較表・出典はこの会話ログを参照（要点のみ以下に記録）。

- ターン制・低頻度更新というこのゲームの特性はSupabase Realtimeの得意分野と一致する
- 「1手ごとの履歴を後で分析・学習に使う」という要件に対し、**関係DB（Postgres/SQL）が最も自然にフィットする**。GCP（Firestore/RTDB）・AWS（DynamoDB）・Azure（Cosmos DB）は標準構成がNoSQL寄りで分析用途に弱い
- GCPの「Firebase SQL Connect」（2026年4月発表）はPostgres統合という点でSupabaseに近いが、発表から数ヶ月の新製品で実績・価格の安定性に懸念がある
- AWS（Amplify Gen2 + Aurora Serverless Postgres）も有力候補だが、**Aurora Postgresの無料枠がアカウント作成から12ヶ月限定かどうか公式情報で確認できず**、期限後は使用量に関わらず課金される可能性がある
- Azure Database for PostgreSQLには無料枠が存在せず、最安構成でも月額$12.41〜が初日から発生する
- **決め手**：Supabaseの無料プランはクレジットカード未登録なら物理的に課金されず、上限超過時は「機能制限（読み取り専用化・API 402エラー等）」という形で安全に倒れる。ブログ経由で想定外のアクセスが来た場合でも、**AWS等の従量課金のように請求が発生するリスクがない**。これが「完全無料前提で進めたい」という方針と合致し、最終的にSupabaseを選定する決め手になった

## セキュリティ・著作権対応

### サイトへのパスワード保護

友人以外がブログ経由でアクセスする可能性を考慮し、**サイト全体に簡易パスワード（アクセスコード）を設置する**。目的は以下の2点：

1. 「本当に友人までにとどめる」という当初からの意図を実効性のあるものにする
2. 想定外のアクセス集中によるSupabase/Vercel無料枠の消費・機能制限を予防する

実装方法（Vercel Middlewareでのベーシック認証 or 簡易パスワード画面等）はフェーズ1の実装時に検討する。

### 著作権に関する一次的見解（非法律家としての整理、正式な法的助言ではない）

- 日本の著作権法は「アイデア・表現二分論」を採用しており、**ゲームのルール・メカニクス自体は「アイデア」として著作権の保護対象外**というのが通説。保護対象になるのはカードの絵柄・ロゴ・具体的なテキスト表現等の「表現」部分
- 現状の設計（オリジナルAI生成絵柄、独自に書き起こしたルール文章、「LAMA」表記を排除し「ビクーニャ」に改名済み、非商用・友人内配布）を前提にすれば、著作権法上の実体的リスクは低いと考えられる
- 残るリスクは「訴訟で負ける」ことより「発行元（AMIGO Spiele等）に見つかり、抗議・削除要請等の対応が発生する」という訴訟未満のトラブル。上記のパスワード保護はこのリスクの低減にも寄与する
- 商用展開（広告収入等）を検討する段階になった場合は、この一次的見解では不十分なため、改めて弁護士等の専門家に確認することを推奨する

## 将来的な検討事項（現時点ではスコープ外）

- **広告収入を得て多数のユーザーに使ってもらいたいという将来的な希望がある**（2026-07-22時点でユーザーから言及）。ただし「まずは今現在」ということで、現フェーズのスコープには含めない
- この将来像は現在の方針（Vercel Hobby＝非商用・広告掲載禁止、友人限定のパスワード保護）と矛盾するため、実現する場合は以下の見直しが必要になる想定：
  - Vercel HobbyからPro（商用利用可）への切替、またはCloudflare Pages等広告掲載可能なホスティングへの移行
  - パスワード保護の解除
  - Supabase無料枠を超える前提でのプラン見直し（Pro化）
  - 著作権面の一次的見解も「非商用」を前提にしているため、商用化する場合は再検討・専門家への確認が必要

## 開発フロー（役割分担）

1. ローカルで開発・テストをClaude Codeと一緒に行う（実装、動作確認、`tsc -b`/`eslint`/Playwright実行まで）
2. `develop`ブランチへのpushは人間側の作業
3. `develop` → `main`へのマージも人間側の作業
4. `main`へのマージ後、Vercelが自動デプロイ（リリース）

Claude Codeはコミット・push・マージを自動で行わない（従来からの方針を継続）。ローカルでの実装・検証まで担当する。
